## MySQL字段类型
### 整数类型的UNSIGNED属性有什么用？
MySQL中的整数类型可以使用可选的UNSIGNED属性来表示不允许负值的无符号整数。使用UNSIGNED属性可以将正整数的上限提高一倍，因为它不需要存储负数值。

### char和varchar的区别？
CHAR

- CHAR类型用于存储固定长度字符串：MySQL总是**根据定义的字符串长度分配足够的空间**。当存储CHAR值时，MySQL会删除字符串中的末尾空格同时，CHAR值需要采用空格进行剩余空间填充，以方便比较和检索。但正因为其长度固定，所以会占据多余的空间，也是一种空间换时间的策略；
- CHAR适合存储很短或长度近似的字符串。例如，CHAR非常适合存储密码的MD5值，定长的身份证号等，因为这些是定长的值
- 对于经常变更的数据，CHAR也比VARCHAR更好，因为定长的CHAR类型占用磁盘的存储的存储空间是连续分配的，不容易产生碎片。
- 对于非常短的列，CHAR也比VARCHAR在存储空间上也更有效率。例如CHAR(1)来存储只有Y和N的值，如果采用单字节字符集只需要一个字节，但是VARCHAR(1)则需要两个字节，因为还有一个记录长度的额外字节

VARCHAR
- VARCHAR类型用于存储可变长度字符串，是最常见的字符串数据类型。它比固定长度类型更节省空间，因为它仅使用必要的空间（根据时间字符串长度改变存储空间）
  

### 为什么不推荐使用TEXT和BLOB？
数据库规范通常不推荐使用TEXT和BLOB数据类型，这两种具有一些缺点和限制，例如：

- 不能有默认值
- 在使用临时表时无法使用内存临时表，只能在磁盘上创建临时表
- 检索效率较低
- 不能直接创建索引，需要指定前缀长度
- 可能导致表上的DML操作过慢

### 为什么不建议使用null作为默认值？
MySQL不建议使用NULL作为默认值，不是因为不能使用索引，而是因为：

- 索引列存在NULL就会导致优化器在做索引选择的时候更加复杂，更加难以优化。比如进行索引统计时，count(1),max(),min()会省略值为NULL的行
- NULL值是一个没意思的值，但是它会占用物理空间，所以会带来的存储空间的问题，因为InnoDB存储记录的时候，如果表中存在允许为NULL的字段，那么行格式（opens new window）中至少会用1字节空间存储NULL值列表。建议用""或0来代替NULL

不建议使用NULL作为默认值，并且建议必须设置默认值

## MySQL基本架构

MySQL主要分为Server层和存储引擎层

- Server层：主要包括连接器、查询缓存、分析器、优化器、执行器等，所有跨存储引擎的功能都在这一层实现，比如存储过程、触发器、视图，函数等，还有一个通用的日志模块binlog日志模块。
- 存储引擎：主要负责数据的存储和读取。server层通过api与存储引擎进行通信。

MySQL Server层主要由下面几部分组成：

- 连接器：建立连接，管理连接、校验用户身份。
- 查询缓存：查询语句如果命中查询缓存则直接返回，否则继续往下执行。MySQL8.0已删除该模块；
- 解析SQL,通过解析器对SQL查询语句进行词法分析、语法分析，然后构建语法树，方便后续模块读取表名、字段、语句类型。
- 执行SQL: 执行SQL共有三个阶段：
  - 预处理阶段：检查表或字段是否存在；将select * 中的*符号扩展为表上的所有列。
  - 优化阶段：基于查询成本的考虑，选择查询成本最小的执行计划（选择使用哪个索引）
  - 执行阶段：根据执行计划执行SQL查询语句，从存储引擎读取记录，返回客户端。

## 存储引擎

### 常用的存储引擎有哪些？

MySQL中常用的四种存储引擎分别是：MyISAM、InnoDB、MEMORY、ARCHIVE。MySQL5.5版本开始，MySQL默认的存储引擎是InnoDB。

**InnoDB存储引擎**
InnoDB是MySQL默认的事务型存储引擎，使用最广泛，基于聚簇索引建立的。InnoDB内部做了很多优化，如能够在内存中创建自适应hash索引，以加速读操作。

**优点**：支持事务和崩溃修复能力；引入了行级锁和外键约束。
**缺点**：占用的数据空间相对较大
**适用场景**：需要事务支持，并且有较高的并发读写频率。

**MyISAM存储引擎**
数据以紧密格式存储。对于只读数据，或者表比较小、可以容忍修复操作，可以使用MyISAM存储引擎。MyISAM会将表存储在两个文件中，数据文件`（.MYD）和索引文件（.MYI）。

**优点**：占用空间较小，查询速度较快。
**缺点**：不支持事务和行级锁，不支持崩溃后的安全恢复，也不支持外键
**适用场景**：对事务完整性没有要求；表的数据都会只读的

**MEMORY存储引擎**
MEMORY存储引擎将数据存储在内存中，因此访问速度非常快。适用于需要快速访问数据的场景，如临时表、缓存等。但是由于数据存储在内存中，所以在服务器重启后数据会丢失。
MEMORY存储引擎使用哈希索引，适合于等值查询，但不适合范围查询。

**优点**：访问速度快，适合临时数据存储。
**缺点**：
1. 哈希索引数据不是按照索引值顺序存储，无法用于排序
2. 不支持部分索引匹配查找
3. 不支持范围查询
4. 当出现哈希冲突时，存储引擎需要遍历链表中所有的行指针，逐行进行比较，直到找到符合条件的行。

**ARCHIVE存储引擎**
ARCHIVE存储引擎适用于存储大量的历史数据，主要用于归档和日志记录等场景。ARCHIVE存储引擎使用压缩格式存储数据，因此可以节省磁盘空间，但查询性能较差。

## MySQL语句
### SQL中select、from、join、where、group by、having、order by、limit的执行顺序是什么？
SQL语句的执行顺序如下：
1. FROM和JOIN：首先处理FROM子句，确定查询涉及哪些表，并执行JOIN操作以连接这些表。
2. join on/union：用于连接多表数据
3. WHERE：对连接后的结果进行过滤，保留满足条件的行。
4. GROUP BY：将结果集按照指定的列进行分组。
5. 有聚合函数时，执行聚合函数计算每个分组的结果。
6. HAVING：对分组后的结果进行过滤，保留满足条件的组。
7. SELECT：选择要返回的列，并执行任何必要的计算。
8. distinct：去重
9. ORDER BY：对结果集进行排序。
10. LIMIT：限制返回的结果行数。

## 索引
### 什么是索引
索引是存储引擎用于提高数据库表的访问速度数据结构。它可以比作一本字典的目录，可以帮你快速找到对应的记录。

索引一般存储在磁盘的文件中，它是占用物理空间的。

### 索引的优缺点
优点：

- 加快数据查找的速度
- 为用来排序或者是分组的字段添加索引，可以加快分组和排序的速度
- 加快表与表之间的连接

缺点：

- 占用磁盘空间：索引需要占用额外的磁盘空间来存储索引数据结构。
- 会降低表的增删改的效率，因为每次对表进行增删改操作时，索引也需要进行相应的更新。

### 索引的作用？
数据是存储在磁盘上的，查询数据时，如果没有索引，会加载所有的数据到内存，依次进行检索，读取磁盘次数较多。有了索引就不需要加载所有数据，因为B+树的高度一般在2-4层，查询时只需要读取2-4次磁盘就可以找到对应的记录，大大减少了磁盘IO的次数，提高了查询效率。

### 什么情况下需要建索引？
1. 经常被查询的字段,即对经常出现在where子句中的列创建索引
2. 经常用于连接的字段，可以加快连接的速度
3. 经常需要排序的字段建立索引
